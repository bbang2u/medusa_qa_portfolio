# 테스트 케이스 명세서 — Medusa QA 포트폴리오

설계·수동 실행한 16개 케이스의 전체 명세(11필드). 각 케이스는 "왜 이 케이스인가"(설계 의도)를 포함한다.
결함으로 이어진 케이스는 해당 결함 ID를 표기하며, 결함 전체 리포트는 [포트폴리오 PDF](Medusa_QA_포트폴리오_김정석.pdf) 부록 A에 수록.

> **요약:** 16건 설계·수동 실행 → 11건 자동화 선별(Playwright 16 테스트) · 5건 자동화 제외(근거 명시). 결함 5건(DEF-01~05) 발견.

## TC-BROWSE-01 · 상점 페이지 가격 정렬 정확성

| 필드 | 내용 |
| :--- | :--- |
| **전제조건** | 백엔드/스토어프론트 기동, 가격 분포가 다양한 상품 데이터 존재 |
| **절차** | 1. `/dk/store?sortBy=price_asc` 진입 2. 표시 가격을 순서대로 수집 3. `price_desc`로 반복 4. 각 정렬이 단조 증가/감소인지 확인 |
| **테스트 데이터** | 기존 상품 데이터(가격 분포 다양) |
| **기대 결과** | 오름차순·내림차순 모두 표시 가격이 정렬 기준대로 단조 증가/감소. 가격 없는·0원 상품도 누락 없이 일관된 위치에 정렬 |
| **설계 의도** | 헤드리스 커머스는 가격이 variant 단위라 "대표 가격"을 무엇으로 잡느냐에 따라 정렬이 깨질 수 있다. 특히 가격이 없거나 0원인 상품이 누락·오정렬되는 결함이 잦다. 정렬이 틀리면 사용자가 최저가 상품을 탐색하지 못해 구매 전환에 직접 영향을 준다 |
| **기법** | 상태 전이(정렬 전후 비교) + 경계값/예외(가격 없는·0원 상품) |
| **우선순위** | Med (구매를 차단하진 않으나 탐색 경험·전환에 영향) |
| **자동화** | O — 결과 결정적, 단언 명확(가격 배열이 정렬됐는지 코드 한 줄로 검증). 사람 눈으로 대조하는 것보다 코드가 더 정확 |
| **실행 결과** | **Pass.** 오름차순·내림차순 모두 정렬 기준대로 확인. 가격 없는/0원 상품도 일관 정렬. (browse-sort.spec.ts와 결과 일치) |

## TC-BROWSE-02 · 카테고리 필터의 상품 격리

| 필드 | 내용 |
| :--- | :--- |
| **전제조건** | 백엔드/스토어프론트 기동, 시드 데이터(카테고리별 1개, ssss는 미지정) 존재 |
| **절차** | 1. `/dk/categories/shirts` 진입 2. 표시 상품 확인 3. (교차) `/dk/categories/sweatshirts` 진입해 다른 상품 확인 |
| **테스트 데이터** | `/dk/categories/shirts`(대표), 교차용 `/dk/categories/sweatshirts` |
| **기대 결과** | shirts에는 t-shirt만 표시. sweatshirt·pants·merch 및 미지정 상품(ssss)은 미표시. sweatshirts에는 sweatshirt만 표시되어 카테고리 간 격리 확인 |
| **설계 의도** | 카테고리 필터가 부실하면 엉뚱한 상품이 보이거나 미분류 상품이 새어 나온다. 특히 카테고리 없는 ssss가 섞여 나오는지가 경계 조건. "목록이 뜨나"가 아니라 필터링 로직이 데이터를 정확히 격리하는가를 본다. BROWSE-01이 순서를 봤다면 이건 분류의 정확성을 본다 |
| **기법** | 동등 분할(카테고리별 그룹) + 경계값(미지정 상품 노출 여부) |
| **우선순위** | Med (오분류 시 탐색 경험 저하, 즉각 차단은 아님) |
| **자동화** | O — 카테고리 구성이 시드로 고정돼 결정적, 표시 상품 개수·이름으로 단언 명확, 카탈로그 구조 변경 시 회귀로 잡을 가치 |
| **실행 결과** | **Pass.** |

## TC-CART-02 · 장바구니 수량 경계값 처리

| 필드 | 내용 |
| :--- | :--- |
| **전제조건** | 백엔드/스토어프론트 기동, 재고 한정 상품 존재 |
| **절차** | 1. 상품을 장바구니에 담기 2. 수량을 0·1·10(UI 상한)·재고+1로 변경 시도 3. 각 경우의 차단/허용·합계 재계산 확인 |
| **테스트 데이터** | 수량 = 0(하한 밖), 1(유효 하한), 10(UI 상한), 재고+1(재고 초과) |
| **기대 결과** | 0·재고 초과는 차단(에러 또는 담기 실패). 유효 범위는 정상 처리되고 합계가 정확히 재계산 |
| **설계 의도** | 수량은 결제 금액과 직결되는 입력이라 경계 처리가 중요하다. 0·재고초과는 결함이 가장 밀집하는 구간으로, 0개나 재고 없는 수량이 담기면 잘못된 주문이 성립해 Blocker로 번진다. UI 선택 상한(10)이 재고와 연동되지 않는 점도 확인 — 재고 3개 상품도 10까지 선택지가 노출되어 선택 후 담기 실패 경험이 발생(차단은 되므로 잘못된 주문은 아님, 사용성 개선 여지) |
| **기법** | 경계값 분석(0·1·10·재고±1) + 예외 경로(재고 초과 시도) |
| **우선순위** | High (수량은 결제 금액·주문 정확성에 직접 영향. 현재 동작은 적절히 차단) |
| **자동화** | O — 핵심 구매경로상 회귀가치 높음, 결과 결정적, 단언 명확(차단/에러/성공 상태를 코드로 검증) |
| **실행 결과** | **Pass.** 0 차단·에러 메시지·재고치 담기 모두 정상 확인 |

## TC-CART-03 · 빈 장바구니 체크아웃 접근 차단

| 필드 | 내용 |
| :--- | :--- |
| **전제조건** | 백엔드/스토어프론트 기동, 장바구니 빈 상태(상품 0개) |
| **절차** | 1. 빈 장바구니(`/dk/cart`) 진입 → 체크아웃 버튼 노출 여부 확인 2. `/dk/checkout` 직접 입력 접근 시도 → 결과 확인 |
| **테스트 데이터** | 없음 (빈 카트 상태 자체가 조건) |
| **기대 결과** | 빈 카트: 체크아웃 버튼 미노출, "비어 있음" 안내 + 탐색 링크. `/dk/checkout` 직접 접근: 체크아웃 미표시·차단 |
| **설계 의도** | 빈 장바구니로 결제 단계에 진입하면 금액 0원 주문 등 비정상 주문이 발생할 수 있다. UI에서 버튼을 숨기는 것만으로는 불충분하며, URL 직접 접근까지 차단되어야 실질적 방어다. UI 차단과 URL 차단을 모두 검증한다 |
| **기법** | 예외 처리/부정 테스트 — 비정상 경로 진입 차단 |
| **우선순위** | Med (비정상 주문 방지 관점에선 중요하나 구조적으로 차단돼 리스크 낮음) |
| **자동화** | X (제외) — 라우팅 구조상 결정돼 회귀가치 낮고, 현재 404 차단이 개선 여지가 있어 자동화로 고정하면 향후 UX 개선을 방해. 수동 확인으로 충분 |
| **실행 결과** | **Pass(관찰).** 빈 카트 버튼 미노출·"Explore products" 링크만 ✅ / `/dk/checkout` 직접 접근 시 404·복귀 경로 제공 ✅. *관찰: 404보다 빈 장바구니 리다이렉트나 안내가 UX상 더 적절(개선 제안, Trivial)* |

| **절차** | 1. t-shirt(€10) 담기 2. ssss(€1) 추가 담기 3. `/dk/cart`에서 라인 2개 금액과 Subtotal 확인 |

| **자동화** | O — 소계 합산은 반드시 지킬 계산 로직, cart-subtotal의 data-value로 단언 결정적·명확. CART-02 헬퍼 재활용 |

## TC-CART-05 · 항목 삭제 후 소계 재계산

| 필드 | 내용 |
| :--- | :--- |
| **전제조건** | 백엔드/스토어프론트 기동, €10 상품 2종이 담긴 장바구니 |
| **절차** | 1. €10 상품 2종 담기(Subtotal 20) 2. 첫 라인 삭제 3. 라인 수·Subtotal 재계산 확인 |
| **테스트 데이터** | t-shirt(€10), sweatshirt(€10) |
| **기대 결과** | 삭제 후 라인 1개, Subtotal 20→10으로 재계산. 남은 라인 합계 = Subtotal |
| **설계 의도** | 담기·수량변경·다중합산에 이어 삭제까지 장바구니 상태 생애주기를 닫는다. 삭제 후 소계가 즉시 정확히 재계산되지 않으면 사용자가 삭제한 상품 값까지 결제할 수 있다. 라인 2→1 경계 전이를 본다 |
| **기법** | 상태 전이(삭제 후 재계산) + 경계값(2→1) |
| **우선순위** | High (금액 정확성 직결) |
| **자동화** | O — 시간이 아니라 "Subtotal이 20→10으로 바뀔 때까지" 상태 델타를 기다리는 방식으로 단언. 삭제 버튼은 testid가 없어 행 내 유일 button으로 선택(취약성 문서화) |
| **실행 결과** | **Pass.** |

| **절차** | 1. 상품 담고 체크아웃 진입 2. 필수 필드 정상 입력 3. 전화번호에 100자 숫자·문자·특수문자 입력 4. 주문 진행 5. Admin에서 저장값 확인 |

| **우선순위** | Medium (선택 필드, 핵심 결제 플로우 차단은 아니나 데이터 무결성에 영향) |
| **자동화** | X (수동 유지) — 입력 검증은 회귀 빈도가 낮고, 의미 있는 검증(DB 영속화 확인)은 Admin 경로까지 필요해 자동화 비용 대비 효용 낮음. 1회 수동으로 DEF-01 식별 완료 |
| **실행 결과** | **Fail.** 100자·문자·특수문자 모두 검증 없이 통과, 주문 완료 및 Phone 필드 영속 저장 확인 → **DEF-01** |

## TC-AUTH-01 · 로그인 인증 결과 및 에러 메시지 일관성

| 필드 | 내용 |
| :--- | :--- |
| **전제조건** | 백엔드/스토어프론트 기동, 유효 계정 보유 |
| **절차** | 이메일 유효성 × 비번 일치 2×2 조합으로 로그인 시도, 각 결과와 실패 메시지 확인 |
| **테스트 데이터** | 가입 이메일+틀린 비번 / 미가입 이메일+임의 비번(2종) |
| **기대 결과** | 실패 케이스는 원인과 무관하게 동일 메시지("Invalid email or password"). "없는 계정"과 "비번 틀림"이 메시지로 구분되지 않아야 함 |
| **설계 의도** | 실패 메시지가 원인별로 다르면 공격자가 "이 이메일은 가입돼 있다"를 메시지로 식별(계정 열거)할 수 있다. 로그인은 가입 폼과 달리 generic 메시지가 베스트프랙티스라, 메시지 일관성으로 열거 방지가 되는지를 본다 |
