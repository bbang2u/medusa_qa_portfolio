TC-BROWSE-01 / 상점 페이지 가격 정렬 정확성 검증 (예외 데이터 포함)
1. TC-ID: TC-BROWSE-01
2. 제목: 상점 페이지 가격 오름차순/내림차순 정렬 정확성 검증
3. 전제조건:

seed 데이터 적용됨
가격이 서로 다른 상품이 2개 이상 존재
(예외 검증용) 가격이 없거나 0원인 상품, 또는 variant별 가격이 다른 상품이 1개 이상 존재

4. 절차:

스토어(:8000) 상점 페이지 진입
정렬 옵션을 "가격 오름차순"으로 선택
표시된 상품들의 가격을 앞에서부터 순서대로 확인
정렬 옵션을 "가격 내림차순"으로 변경
표시된 상품들의 가격 순서를 다시 확인
(예외) 가격 없는/0원/variant 상품이 정렬 목록에서 어디에 위치하는지 확인

5. 테스트 데이터: 해당 없음 (기존 상품 데이터 사용. 단 가격 분포가 다양해야 함)
6. 기대 결과:

오름차순: 표시된 상품 가격이 앞→뒤로 갈수록 작아지지 않는다 (각 상품 가격 ≤ 바로 다음 상품 가격). 첫 상품이 최저가.
내림차순: 그 반대 (각 상품 가격 ≥ 다음 상품 가격). 첫 상품이 최고가.
가격 없는/0원 상품: 누락되지 않고, 일관된 위치(맨 앞 또는 맨 뒤)에 정렬된다. 정렬 기준에 따라 위치가 예측 가능해야 한다.

7. 설계 의도:
정렬은 흔한 기능이지만, 헤드리스 커머스는 가격이 variant 단위라 "대표 가격"을 무엇으로 잡느냐(최저가/첫 variant 등)에 따라 정렬이 깨질 수 있다. 특히 가격이 없거나 0원인 상품이 정렬에서 누락되거나 엉뚱한 위치로 가는 결함이 실제로 자주 발생한다. 정렬이 틀리면 사용자가 최저가 상품을 탐색하지 못해 구매 전환에 직접 영향을 준다.
8. 기법: 상태 전이(정렬 상태 변경 전후 비교) + 경계값/예외(가격 없는·0원 상품)
9. 우선순위: Med (구매를 차단하진 않으나 탐색 경험·전환에 영향)
10. 자동화: O — 결과 결정적(동일 데이터면 동일 순서), 단언 명확(가격 배열이 정렬됐는지 코드로 한 줄 검증). 사람 눈으로 가격 대조하는 것보다 코드가 더 정확한 케이스.
11. 실행결과: (손실행 후 기입)

TC-CART-02 / 장바구니 수량 경계값 처리 검증
1. TC-ID: TC-CART-02
2. 제목: 장바구니 수량 변경 시 경계값(하한·상한·재고초과) 처리 검증
3. 전제조건:

seed 데이터 적용됨
재고가 있는 상품이 장바구니에 1개 담겨 있음
재고 수량이 다른 상품들 확보 (재고 10 이상 / 10 미만 각 1개 — 상한 연동 확인용)

4. 절차:

장바구니에서 수량 선택 UI 확인 (선택 가능 범위)
수량을 1로 설정 (유효 하한)
수량을 0으로 내릴 수 있는지 확인
수량을 최대 선택값(10)으로 설정
재고가 10 미만인 상품에서 재고보다 큰 수량 선택 시도
재고와 정확히 같은 수량 담기 시도 (상한 경계)

5. 테스트 데이터: 수량 = 0(하한 밖), 1(유효 하한), 10(UI 상한), 재고+1(재고 초과)
6. 기대 결과:

수량 선택 범위는 1~10이며, 0은 선택 불가 (하한이 UI에서 차단됨)
수량 0으로 만들려면 별도 "삭제" 버튼을 사용해야 함 (수량 변경과 삭제가 분리됨)
선택 상한값 10은 재고와 무관한 UI 고정값 (재고 3개인 상품도 10까지 선택지 노출)
재고를 초과하는 수량으로 담기 시도 시 명확한 에러 메시지와 함께 담기 실패 (재고 없는 수량이 담기지 않음)
재고와 정확히 같은 수량은 정상적으로 담긴다 (상한 경계 off-by-one 없음)

7. 설계 의도:
수량은 결제 금액과 직결되는 입력이라 경계 처리가 중요하다. 0(하한)·재고초과(상한)는 결함이 가장 밀집하는 구간으로, 0개 항목이 담기거나 재고 없는 수량이 담기면 잘못된 주문이 성립해 Blocker로 번진다. 추가로 UI 선택 상한(10)이 재고와 연동되지 않는 점을 확인 — 재고 3개 상품도 10까지 선택지가 노출되어, 사용자가 선택은 했으나 담기에서 실패하는 경험이 발생한다. 차단은 되므로 잘못된 주문으로 이어지진 않으나, 선택지를 재고에 맞게 제한하면 사용성이 개선될 여지가 있다.
8. 기법: 경계값 분석(0·1·10·재고±1) + 예외 경로(재고 초과 시도)
9. 우선순위: High (수량은 결제 금액·주문 정확성에 직접 영향. 단 현재 동작은 적절히 차단됨)
10. 자동화: O — 핵심 구매경로상 회귀가치 높음, 결과 결정적(동일 입력 동일 결과), 단언 명확(차단/에러/성공 상태를 코드로 검증 가능)
11. 실행결과: (손실행 후 기입 — 현재까지 손확인: 0 차단·에러메시지·재고치 담기 모두 정상)


TC-CHECKOUT-01 / 체크아웃 전화번호 필드 입력 검증
1. TC-ID: TC-CHECKOUT-01
2. 제목: 체크아웃 전화번호 필드의 비정상 입력(길이·타입) 검증
3. 전제조건:

seed 데이터 적용됨
장바구니에 상품 1개 이상 담긴 상태
체크아웃 진입 가능, 이메일 등 필수 필드는 정상 입력 가능

4. 절차:

상품을 장바구니에 담고 체크아웃 진입
필수 필드(이메일·이름·주소 등)는 유효값으로 입력
전화번호 필드에 무효값을 입력: ① 숫자 100자 이상 ② 문자(abcdef) ③ 특수문자(!@#$%)
주문을 끝까지 진행
Admin → Customers → 해당 고객 상세에서 Phone 필드 저장 상태 확인

5. 테스트 데이터:

정상: 01012345678
무효: 2×100자(길이 초과) / abcdef(문자) / !@#$%^&(특수문자)

6. 기대 결과:

전화번호 필드는 숫자 외 문자·특수문자를 거부하거나, 합리적 최대 길이(국제번호 기준 약 15~20자)로 제한해야 한다
비정상 형식 입력 시 제출 시점에 검증 에러가 표시되고 주문 진행이 차단되어야 한다
(선택 필드이므로 빈값 통과는 정상 동작으로 간주)

7. 설계 의도:
전화번호는 배송·CS 단계의 핵심 연락 수단이다. 국가별 형식 차이로 느슨하게 두는 경우가 있으나, 문자·특수문자·과도한 길이까지 무검증으로 허용하면 비정상 데이터가 주문과 함께 고객 DB에 영속 저장된다. 입력 검증의 경계(어디까지 허용되는가)를 확인하여 데이터 무결성 리스크를 식별한다.
8. 기법: 입력 검증 + 경계값 분석(길이) + 동등 분할(숫자/문자/특수문자 그룹)
9. 우선순위: Medium (선택 필드이고 핵심 결제 플로우를 차단하진 않으나, 데이터 무결성에 영향)
10. 자동화: X (수동 유지) — 입력 검증은 회귀 빈도가 낮고, 의미 있는 검증(DB 영속화 확인)은 Admin 경로까지 필요해 자동화 비용 대비 효용이 낮음. 1회 수동 검증으로 결함(DEF-01) 식별을 완료했으므로 수동으로 남김.
11. 실행결과: Fail — 숫자 100자·문자·특수문자 모두 검증 없이 통과, 주문 완료 및 고객 Phone 필드 영속 저장 확인. → DEF-01 참조

7. 설계 의도:

로그인은 인증의 1차 관문이다. 실패 메시지가 "없는 계정"과 "비번 틀림"을 구분하면, 공격자가 메시지 차이로 특정 이메일의 가입 여부를 알아내는 계정 열거(account enumeration)가 가능해진다. 모든 실패를 동일 메시지로 처리하는지 검증하여 인증 단계의 정보 노출 리스크를 식별한다.
8. 기법: 결정 테이블 테스트(이메일 유효성 × 비번 일치 2×2 조합) + 보안 관점(정보 노출 검증)
9. 우선순위: Medium (핵심 보안 관문이나, 견고할 경우 결함 아님 — 리스크 식별 목적)
10. 자동화: O — 입력→결과 패턴이 결정적이고, 4개 조합 단언이 명확. 단 성공 케이스(1번)는 실제 비번이 코드에 노출되므로 자동화 시 실패 3건(메시지 일관성)만 자동화하고 성공은 수동 또는 환경변수 처리 권장.
11. 실행 결과: Pass — 4개 조합 모두 기대대로 동작. 실패 3건 메시지 전부 "Invalid email or password"로 동일. 계정 열거 방지 확인. (견고함 확인 사례)
